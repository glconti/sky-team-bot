## Work Completed

### 1. Callback Query Handling Architecture
- **Decision:** Inline keyboard buttons → callback queries → action handler
- **Handler pattern:** `HandleCallbackQueryAsync(callbackQuery)` validates payload, executes action, sends `AnswerCallbackQuery` response
- **Idempotency:** track `(userId, groupChatId, menuVersion, actionId)` to dedupe retries
- **Expiry:** stale button clicks (menu version mismatch) → toast "menu expired, refresh" + no-op

### 2. Single Edited Cockpit Message Pattern
- **Group cockpit:** on first interaction (`/sky new` or "Open" button), send initial cockpit message, persist `message_id`
- **All subsequent updates:** `EditMessageText` with updated keyboard + text (no new message spam)
- **Cockpit lifecycle:** sent at session start, edited after each round resolution, deleted on game end or session timeout

### 3. DM Menu (Hand + Commands)
- **Trigger:** on `/start` or on-demand `/sky hand` or `/sky place`
- **Content:** secret dice hand display + available placement commands (as inline buttons)
- **Message persistence:** store DM `message_id` per user; edit on state changes (die used, undo, new round)
- **Buttons:** die-selector buttons (0–3) + target buttons (module slots) + undo/cancel

### 4. callback_data 64-byte Constraint
- **Problem:** Full command IDs + dynamic options exceed 64 bytes
- **Solution:** short versioned action tokens + server-side state lookup
- **Format:** `v1:action:index` (e.g., `v1:place:d2` = "place die at index 2")
- **Mapping:** `MenuState[(userId, groupChatId, menuVersion)]` → `{ actionId → CommandId, targetId → ModuleSlot }`

### 5. Menu State Store Design
- **Storage:** in-memory `Dictionary<(string userId, long groupChatId, int menuVersion), MenuState>`
- **MenuState:** `{ ActionId → CommandId, TargetId → ModuleSlot, Timestamp }`
- **Lifecycle:** created on menu render, garbage collected on session end or 1-hour timeout
- **Concurrency:** thread-safe with `ReaderWriterLockSlim` per group (serialize Telegram updates)

### 6. Deep-Link Onboarding Flow
- **Entry:** `/start?game=<groupId>` (from group message button or manual invite)
- **Handler:** `HandleStartAsync(message, groupId)` → register user, join group session, show DM menu
- **Fallback:** plain `/start` (no groupId) → list active games or "no active games" message

### 7. Technical Constraints & Mitigations
- **Constraint:** Telegram callback retry may deliver updates out-of-order
  - **Mitigation:** each handler validates `menuVersion` and idempotency key before executing
- **Constraint:** bot restart loses in-memory menu state
  - **Mitigation:** stale buttons → `AnswerCallbackQuery` with "menu expired" toast; user retries from `/sky hand`
- **Constraint:** high concurrency (multiple groups, players, message edits)
  - **Mitigation:** per-group serialization + async/await on all Telegram API calls
